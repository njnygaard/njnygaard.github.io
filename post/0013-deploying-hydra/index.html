Deploying Hydra

07 Jan 2018, 11:00

I will document how I deploy Hydra here. We use Ansible to deploy to AWS so most of this will be high-level about how we deploy to an EC2 instance.

As of writing, the latest Hydra image is v0.10.10. I will be using that for the tutorial.

Compressed Steps (mac)

I needed to get this working on mac because that is one of our development environments. The next step is to get it running on production.

echo "create database hydra" | mysql -uroot -proot

docker run -it --rm \
  oryd/hydra:v0.10.10 \
  migrate sql mysql://root:root@tcp(docker.for.mac.localhost:3306)/hydra?parseTime=true

docker run -d \
  --name ory-hydra-example--hydra \
  -p 9009:4444 \
  -e SYSTEM_SECRET=yJFLU44byGmmKLwJHvramNknAmSQR27C \
  -e DATABASE_URL=mysql://root:root@tcp(docker.for.mac.localhost:3306)/hydra?parseTime=true \
  -e ISSUER=https://localhost:9009/ \
  -e CONSENT_URL=http://localhost:9020/consent \
  -e FORCE_ROOT_CLIENT_CREDENTIALS=admin:demo-password \
  oryd/hydra:v0.10.10

This works for the local installation which will get the test environment working. But this is not even close to what I need to get it to work in a production environment. I have all kinds of ports to consider.

Ansible Flow

I don’t want to document all the steps because the playbook is long. I’ll list the high level infrastructure steps here:

  • Create Hydra Database
    • You have to create a database in your DB of choice so Hydra can save its data between startups.
  • Create the Hydra Network
    • You need to create a docker network so that it is easy for the consent app to talk to the Hydra instance.
  • Run Hydra Migrations
    • This sets up the tables for Hydra in the database you just created.
  • Start Hydra
    • Start the docker container for Hydra.
  • Create Client for Consent App
    • This basically informs hydra that there is an application out there that is allowed to connect to it.
    • When you create the client, you create a port that will be passed as the redirect uri.
  • Create Policy for Consent App
    • TODO I’m still not clear on what the policy does.
  • Run Consent App
    • Now we run a docker container with the consent app that knows that its HYDRA_CLIENT_ID is consent-app.

Sample Flow

Once you have all your infrastructure working, you can test it on the EC2 like so:

Create Client

Up

sudo docker run --rm -it \
  -e CLUSTER_URL=https://hydra:4444 \
  -e CLIENT_ID=admin \
  -e CLIENT_SECRET=demo-password \
  --network hydra \
  oryd/hydra:v0.10.10 \
  clients create --skip-tls-verify \
    --id some-consumer \
    --secret consumer-secret \
    --grant-types authorization_code,refresh_token,client_credentials,implicit \
    --response-types token,code,id_token \
    --allowed-scopes openid,offline,hydra.clients \
    --callbacks http://localhost:10010/callback

Down

sudo docker run --rm -it \
  -e CLUSTER_URL=https://hydra:4444 \
  -e CLIENT_ID=admin \
  -e CLIENT_SECRET=demo-password \
  --network hydra \
  oryd/hydra:v0.10.10 \
  clients delete some-consumer --skip-tls-verify

Create Policy

Up

sudo docker run --rm -it \
  -e CLUSTER_URL=https://hydra:4444 \
  -e CLIENT_ID=admin \
  -e CLIENT_SECRET=demo-password \
  --network hydra \
  oryd/hydra:v0.10.10 \
  policies create --skip-tls-verify \
    --actions get \
    --description "Allow everyone to read the OpenID Connect ID Token public key" \
    --allow \
    --id openid-id_token-policy \
    --resources rn:hydra:keys:hydra.openid.id-token:public \
    --subjects "<.*>"

Down

sudo docker run --rm -it \
  -e CLUSTER_URL=https://hydra:4444 \
  -e CLIENT_ID=admin \
  -e CLIENT_SECRET=demo-password \
  --network hydra \
  oryd/hydra:v0.10.10 \
  policies delete openid-id_token-policy --skip-tls-verify

Create Token

Up

sudo docker run --rm -it \
  --network hydra \
  -p 10010:4445 \
  oryd/hydra:v0.10.10 \
  token user --skip-tls-verify \
    --auth-url https://localhost:10000/oauth2/auth \
    --token-url https://hydra:4444/oauth2/token \
    --id some-consumer \
    --secret consumer-secret \
    --scopes openid,offline,hydra.clients \
    --redirect http://localhost:10010/callback

Down
